This guide explains how to pick the right installer command and what Outfitting changes on your machine. Use this page when you want to understand the model, not just copy a single command.

## How to use

1. Pick your platform: Windows, WSL, or macOS.
2. Pick scope: minimal/bootstrap, default personal setup, or full setup.
3. Pick profile set:
   - Windows uses explicit profiles like `base`, `dev`, `work`, `gaming`, `qol`, `network`.
   - WSL supports personal or work profile flags.
   - macOS uses Homebrew casks and formulae, and the repo's nix-darwin/Home Manager config.
4. Run install command as documented in the platform page.
5. Optionally run the platform post-install command to fetch the Cloudflare
   Access-protected licensed fonts. Windows extracts them to the user's
   Downloads folder; macOS installs them system-wide.
6. Verify by running profile/update helper commands (`hm-profile`, `hm-sync`, `hm-update`, etc.).

## What gets installed

Outfitting installs and configures four layers:

### 1) Package Layer

- **Windows**:
  - WinGet profiles (`base`, `dev`, etc.).
  - Optional Microsoft Store profiles (`msstore-*`).
  - Optional Bun global packages via `win.jfa.dev/bun`.
- **WSL**:
  - Nix/Home Manager packages from `packages/x64-linux`.
  - Optional APT packages in update/full modes.
- **macOS**:
  - Homebrew casks and formulae from `packages/aarch64-darwin/Brewfile`.
  - nix-darwin/Home Manager configuration from `packages/aarch64-darwin`.

### 2) Configuration Layer

- Dotfiles and shell profiles from this repository.
- Home Manager configuration through symlinks.
- PowerShell profile updates on Windows via `win.jfa.dev/config/pwsh-profile`.

### 3) Environment Wiring

- Repository is cloned into `~/.config/outfitting/repo`.
- Symlinks connect your platform config paths to repo-managed files.
- On macOS, nix-darwin/Home Manager symlinks are created for rebuild commands, while package manifests stay in the repo for Homebrew..

### 4) Maintenance Tooling

- Helper commands to apply and update configs:
  - `hm-sync`, `hm-update`, `hm-personal`, `hm-work` (WSL)
  - `outfit sync`, `outfit upgrade`, `outfit switch` (macOS)
- Update scripts to refresh package sources and reinstall as needed.

### Optional post-install layer

- **Windows**: `irm win.jfa.dev/post-install | iex`; licensed fonts are
  extracted to `%USERPROFILE%\Downloads\fonts` for optional manual installation,
  while fonts in `fonts/fontget.txt` are installed through FontGet. Missing
  Cloudflare Access permission skips only the optional licensed-font download.
- **macOS**: `curl -L mac.jfa.dev/post-install | bash`, which requests `sudo`
  when installing fonts.
- Both commands use an interactive Cloudflare Access login to download the
  licensed font archive. The archive is checksum-verified before its desktop
  fonts are installed for all local users.

## Command selection examples

### Windows

- First-time development machine:

```powershell
irm win.jfa.dev/base+dev+qol | iex
```

- Add Microsoft Store apps:

```powershell
irm win.jfa.dev/msstore/msstore-base+msstore-dev | iex
```

### WSL

- Default personal setup:

```bash
curl -L wsl.jfa.dev | bash
```

- Full setup with work profile:

```bash
curl -L wsl.jfa.dev | bash -s -- --full-install --work-profile
```

### macOS

- Standard install:

```bash
curl -L mac.jfa.dev | bash
```

## Post-install commands

Run these only after the main installer. They are optional, but required to
install the private licensed-font release.

```powershell
irm win.jfa.dev/post-install | iex
```

```bash
curl -L mac.jfa.dev/post-install | bash
```

## Safety and repeatability

- Installations are profile-driven and intended to be repeatable.
- Re-running installer/update commands should converge your machine toward the repository state.
- Prefer adding/removing packages in this repo, then run sync/update commands rather than making one-off manual changes.
